LAB #01 (APPRENTICE)

SQL injection vulnerability in WHERE clause allowing retrieval of hidden data

Este laboratorio contiene una vulnerabilidad de inyección SQL en el filtro de categoría de productos. Al seleccionar una categoría, la aplicación ejecuta una consulta que filtra por productos lanzados (released = 1). El objetivo es manipular la lógica de la consulta para omitir este filtro y visualizar productos no publicados.

' OR 1=1 --
Metodología: Se interceptó la solicitud con Burp Suite y se modificó el parámetro category. El payload cierra la comilla simple de la consulta original, añade una condición siempre verdadera (OR 1=1) y comenta el resto de la sentencia original (--).
LAB #02 (APPRENTICE)

SQL injection vulnerability allowing login bypass

Este laboratorio presenta una vulnerabilidad de inyección SQL en la función de inicio de sesión. El objetivo es realizar un ataque que permita acceder al sistema como el usuario administrator sin poseer sus credenciales legítimas.

administrator'--
Metodología: Se utilizó Burp Suite para interceptar la solicitud POST del login. Al modificar el parámetro username con el payload, se logra que la consulta SQL considere únicamente el nombre de usuario y comente el resto de la validación (incluyendo la contraseña), forzando el acceso exitoso.
LAB #03 (PRACTITIONER)

SQL injection attack, querying the database type and version on Oracle

Este laboratorio contiene una vulnerabilidad de inyección SQL en el filtro de categorías. El objetivo es utilizar un ataque UNION para recuperar la cadena de versión de la base de datos Oracle.

' UNION SELECT BANNER, NULL FROM v$version--
Metodología: 1. Se determinó el número de columnas (2) y cuáles contienen texto mediante el payload ' UNION SELECT 'abc', 'def' FROM dual--.
2. Al tratarse de Oracle, es obligatorio incluir la cláusula FROM con una tabla válida (como dual o v$version).
3. Se inyectó el payload final para extraer la información del banner de la versión del sistema.
LAB #04 (PRACTITIONER)

SQL injection attack, querying the database type and version on MySQL and Microsoft

Este laboratorio contiene una vulnerabilidad de inyección SQL en el filtro de categorías. El objetivo es recuperar la cadena de versión de la base de datos utilizando un ataque UNION adaptado para MySQL o Microsoft SQL Server.

' UNION SELECT @@version, NULL#
Metodología: 1. Se determinó el número de columnas y cuáles contienen texto mediante el payload ' UNION SELECT 'abc', 'def'#. El uso del carácter # es fundamental para comentar el resto de la consulta en MySQL.
2. A diferencia de Oracle, en MySQL y Microsoft no siempre es necesaria la cláusula FROM para consultas simples de sistema.
3. Se inyectó el payload final utilizando la constante @@version para extraer la información técnica del motor de base de datos.
LAB #05 (PRACTITIONER)

SQL injection attack, listing the database contents on non-Oracle databases

Este laboratorio es una práctica completa de exfiltración de datos. El objetivo es identificar las tablas y columnas de la base de datos para recuperar el nombre de usuario y la contraseña de todas las cuentas, y posteriormente autenticarse como administrator.

' UNION SELECT table_name, NULL FROM information_schema.tables--
' UNION SELECT column_name, NULL FROM information_schema.columns WHERE table_name = 'users_abcdef'--
' UNION SELECT username_abcdef, password_abcdef FROM users_abcdef--
Metodología: 1. Determinación de columnas: Se confirmó que la consulta devuelve dos columnas de texto[cite: 520].
2. Enumeración de tablas: Se consultó information_schema.tables para localizar la tabla de usuarios (ej. users_abcdef)[cite: 520].
3. Enumeración de columnas: Se consultó information_schema.columns para obtener los nombres exactos de los campos de credenciales[cite: 520].
4. Exfiltración: Se ejecutó el UNION SELECT final sobre la tabla específica para obtener la contraseña del administrador[cite: 520].
LAB #06 (PRACTITIONER)

SQL injection attack, listing the database contents on Oracle

Este laboratorio implica la exfiltración de credenciales de usuario desde una base de datos Oracle. El objetivo es navegar por el diccionario de datos de Oracle para localizar la tabla de usuarios y sus columnas correspondientes para obtener la contraseña del usuario administrator.

' UNION SELECT table_name, NULL FROM all_tables--
' UNION SELECT column_name, NULL FROM all_tab_columns WHERE table_name = 'USERS_ABCDEF'--
' UNION SELECT USERNAME_ABCDEF, PASSWORD_ABCDEF FROM USERS_ABCDEF--
Metodología: 1. Prueba de columnas: Se validó el número de columnas y tipos de datos usando ' UNION SELECT 'abc', 'def' FROM dual--.
2. Listado de tablas: Se consultó la tabla de sistema all_tables para identificar la tabla que almacena credenciales.
3. Listado de columnas: Se utilizó all_tab_columns filtrando por el nombre de la tabla descubierta para obtener los nombres de los campos.
4. Extracción: Se ejecutó la consulta final para recuperar el nombre de usuario y contraseña del administrador.
LAB #07 (PRACTITIONER)

SQL injection UNION attack, determining the number of columns returned by the query

Este laboratorio contiene una vulnerabilidad de inyección SQL en el filtro de categorías. El objetivo es determinar el número exacto de columnas devueltas por la consulta original mediante el uso de valores NULL, técnica necesaria para construir ataques UNION más complejos en fases posteriores.

' UNION SELECT NULL--
' UNION SELECT NULL, NULL--
' UNION SELECT NULL, NULL, NULL--
Metodología: 1. Se interceptó la solicitud con Burp Suite y se modificó el parámetro category.
2. Se inyectaron progresivamente valores NULL en la sentencia UNION SELECT.
3. Se observó que la aplicación devolvía un error hasta que el número de NULL coincidió con el número de columnas de la consulta original.
4. La resolución del laboratorio se confirma cuando la aplicación responde correctamente, incluyendo el contenido adicional de los valores nulos.
LAB #08 (PRACTITIONER)

SQL injection UNION attack, finding a column containing text

Este laboratorio requiere identificar cuál de las columnas devueltas por una consulta original acepta datos de tipo texto. El objetivo es hacer que un valor aleatorio proporcionado por el laboratorio aparezca en los resultados de la consulta.

' UNION SELECT 'abcdef', NULL, NULL--
' UNION SELECT NULL, 'abcdef', NULL--
' UNION SELECT NULL, NULL, 'abcdef'--
Metodología: 1. Se utilizó Burp Suite para determinar que la consulta original devuelve tres columnas usando ' UNION SELECT NULL, NULL, NULL--[cite: 17].
2. Se reemplazó sistemáticamente cada valor NULL por la cadena de texto proporcionada (ej. 'abcdef')[cite: 17].
3. Si la base de datos devuelve un error, significa que esa columna no es compatible con texto; si la página carga y muestra el valor, se ha identificado una columna válida[cite: 17].
4. Este proceso se repite hasta encontrar la columna que permite la exfiltración de datos legibles[cite: 17].
LAB #09 (PRACTITIONER)

SQL injection UNION attack, retrieving data from other tables

Este laboratorio requiere la combinación de técnicas de inyección UNION para recuperar datos de una tabla distinta a la de la consulta original. Se debe extraer el contenido de la tabla users, la cual contiene las columnas username y password.

' UNION SELECT username, password FROM users--
Metodología: 1. Análisis inicial: Se utilizó Burp Suite para confirmar que la consulta devuelve dos columnas y que ambas son compatibles con tipos de datos de texto mediante el payload ' UNION SELECT 'abc', 'def'--.
2. Extracción de datos: Al conocer los nombres de la tabla y las columnas por la descripción del reto, se ejecutó un UNION SELECT directo para volcar todas las credenciales.
3. Explotación final: Se localizaron las credenciales del usuario administrator en la respuesta de la aplicación y se procedió con el inicio de sesión exitoso.

SQL injection attack, querying the database type and version on PostgreSQL

Este laboratorio contiene una vulnerabilidad de inyección SQL en el filtro de categorías. El objetivo es extraer la cadena de versión de la base de datos PostgreSQL utilizando un ataque de tipo UNION.

' UNION SELECT version(), NULL--
Metodología: 1. Análisis de columnas: Se determinó que la consulta devuelve dos columnas y que la primera es compatible con tipos de datos de texto mediante el payload ' UNION SELECT 'abc', NULL--.

LAB #11 (PRACTITIONER)

Blind SQL injection with conditional responses

Este laboratorio utiliza una cookie de rastreo para realizar consultas SQL que no devuelven resultados visibles. El objetivo es explotar esta vulnerabilidad ciega para extraer la contraseña del usuario administrator analizando las respuestas condicionales de la página.

TrackingId=xyz' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='administrator')='a
Metodología: 1. Confirmación de Vulnerabilidad: Se inyectaron condiciones booleanas en la cookie TrackingId. La respuesta ' AND '1'='1 mostró un mensaje de "Welcome back", mientras que ' AND '1'='2 lo hizo desaparecer.
2. Enumeración de Longitud: Se utilizó ' AND (SELECT username FROM users WHERE username='administrator' AND LENGTH(password)>19)='administrator para confirmar que la contraseña tiene 20 caracteres.
3. Ataque de Fuerza Bruta: Se configuró Burp Intruder para iterar a través de cada posición de la contraseña (1 a 20) y probar todos los caracteres posibles (a-z, 0-9) usando la función SUBSTRING().
4. Inferencia: Se filtraron los resultados en Burp Suite buscando la presencia del mensaje "Welcome back" para reconstruir la contraseña completa.
LAB #12 (PRACTITIONER)

Blind SQL injection with conditional errors

En este laboratorio, la base de datos no devuelve resultados ni mensajes de error informativos. Sin embargo, se puede inducir un error de servidor (HTTP 500) mediante consultas mal formadas para inferir datos. El objetivo es extraer la contraseña de administrator en una base de datos Oracle.

TrackingId=xyz' || (SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM users WHERE username='administrator') || '
Metodología: 1. Identificación del motor: Se confirmó el uso de Oracle al requerir la cláusula FROM dual en subconsultas para evitar errores de sintaxis.
2. Inferencia mediante errores: Se utilizó la función TO_CHAR(1/0) para forzar una división por cero (error 500) solo cuando una condición CASE específica es verdadera.
3. Extracción carácter por carácter: Se automatizó con Burp Intruder el uso de SUBSTR() para validar cada posición de la contraseña. Si la aplicación devolvía un error 500, el carácter probado era correcto.
4. Validación final: Se determinó que la contraseña tiene 20 caracteres y se reconstruyó mediante la observación de los estados de error del servidor.
LAB #13 (PRACTITIONER)

Visible error-based SQL injection

Este laboratorio utiliza una cookie de rastreo para una consulta SQL cuyos resultados no se muestran. El objetivo es provocar un error deliberado que contenga datos de la base de datos en su descripción. Se debe extraer la contraseña del usuario administrator.

' AND 1=CAST((SELECT password FROM users LIMIT 1) AS int)--
Metodología: 1. Provocación del error: Se insertó una comilla simple en la cookie TrackingId para confirmar que la aplicación devuelve errores detallados del motor de base de datos.
2. Prueba de casting: Se intentó convertir una subconsulta en un tipo de dato entero (int). Al fallar la conversión, la base de datos muestra el contenido original que intentó convertir en el mensaje de error.
3. Fuga de datos (Leak): Se ajustó la consulta para seleccionar el campo username de la tabla users. El error reveló "administrator".
4. Exfiltración final: Se cambió el campo a password. El sistema respondió con un error de sintaxis que incluía la contraseña real: invalid input syntax for type integer: "contraseña_extraída".
LAB #14 (PRACTITIONER)

Blind SQL injection with time delays

Este laboratorio utiliza una cookie de rastreo para una consulta SQL. La aplicación no devuelve resultados ni errores, pero la ejecución es síncrona. El objetivo es confirmar la vulnerabilidad provocando un retardo de 10 segundos en la respuesta del servidor.

TrackingId=x'||pg_sleep(10)--
Metodología: 1. Identificación de Vulnerabilidad Ciega: Se inyectaron diversos payloads sin observar cambios en la interfaz ni mensajes de error.
2. Prueba de Tiempo: Se utilizó la función pg_sleep(10) integrada en la cookie TrackingId.
3. Confirmación: Se monitoreó el tiempo de respuesta en la pestaña de red de Burp Suite, confirmando que el servidor tardó exactamente 10 segundos en procesar la solicitud, lo que valida la inyección en una base de datos PostgreSQL.
LAB #15 (PRACTITIONER)

Blind SQL injection with time delays and information retrieval

Este es el escenario de explotación más sigiloso. La aplicación es totalmente ciega a cambios de contenido y errores. El objetivo es recuperar la contraseña del usuario administrator utilizando consultas que provocan retardos de tiempo condicionales solo cuando una suposición es correcta.

TrackingId=x'%3BSELECT+CASE+WHEN+(username='administrator'+AND+SUBSTRING(password,1,1)='a')+THEN+pg_sleep(10)+ELSE+pg_sleep(0)+END+FROM+users--
Metodología: 1. Inferencia Booleana mediante Tiempo: Se utilizó una sentencia CASE para ejecutar pg_sleep(10) si la condición es verdadera, o pg_sleep(0) si es falsa.
2. Validación de Longitud: Se confirmó que la contraseña tiene 20 caracteres observando retardos de 10 segundos al probar LENGTH(password)=20.
3. Automatización de Exfiltración: Se configuró Burp Intruder con un solo hilo (Maximum concurrent requests = 1) para medir con precisión el tiempo de respuesta de cada carácter probado.
4. Reconstrucción: Aquellas solicitudes que tardaron ~10,000ms revelaron el carácter correcto de la contraseña, permitiendo el acceso final como administrador.
